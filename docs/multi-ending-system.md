# Multi-Ending System

## Purpose

The project direction is a private long-form novel ending lab, not a single-answer ending generator.

The system should generate, compare, hybridize, and revise multiple coherent ending routes for the same long novel. For the first private experiment, the target case is *Dragon Raja / Long Zu*, framed as a private creative research target for multi-ending design, author-style consistency, story continuity, and emotional core preservation.

## Why multiple endings instead of one final answer

A long novel ending is not only a plot answer. It is a design decision across theme, character payoff, worldbuilding closure, emotional rhythm, and reader expectation. A single generated ending can look confident while hiding weak assumptions.

Multiple endings make the system stronger because they force it to:

1. Explore different valid interpretations of the same story memory.
2. Separate story logic from emotional preference.
3. Show how the same unresolved threads can be resolved in different ways.
4. Reveal tradeoffs between closure, tragedy, scale, ambiguity, and comfort.
5. Give the user a planning space before full prose drafting begins.

## How multiple endings prove creative reasoning

A multi-ending workflow demonstrates creative reasoning better than one polished answer because the system must make structured choices and explain them.

The system should prove that it can:

- Generate distinct routes without collapsing them into the same plot.
- Keep every route consistent with the same story bible.
- Track unresolved plotlines and foreshadowing across alternate resolutions.
- Explain why a complete closure ending feels different from a tragic fate ending.
- Compare emotional effect, logic, worldbuilding fit, and expansion potential.
- Merge strong parts of different routes into a hybrid route.
- Identify risks before chapter drafting begins.

## Required ending route families

The system should support at least five base route types:

| Code | Route type | Main purpose |
| --- | --- | --- |
| A | Complete closure ending | Resolve major conflicts and give strong narrative finality. |
| B | Tragic fate ending | Emphasize destiny, sacrifice, loss, and irreversible cost. |
| C | Epic battle ending | Pay off large-scale conflict through decisive confrontation. |
| D | Open resonance ending | Leave selected questions open while creating thematic afterglow. |
| E | Emotional compensation ending | Repair emotional wounds and give characters intimate payoff. |

## Hybrid route support

The system should also support mixed endings when one route type is too narrow.

Required hybrid examples:

- A + B: complete closure with tragic cost
- B + D: tragic fate with open resonance
- A + C: complete closure through epic confrontation
- D + E: open resonance with emotional compensation
- User-custom mixed route: any user-defined blend of route priorities

## First deliverable: ending outlines

The first milestone is not to write final chapters. The first deliverable is a set of high-quality ending outlines that can be compared and revised.

Each route should include:

1. Route name
2. Core theme
3. Emotional tone
4. Main conflict resolution
5. Protagonist final arc
6. Major character endings
7. Worldbuilding resolution
8. Unresolved plotline recovery
9. Foreshadowing recovery
10. Final volume structure
11. Chapter-by-chapter outline
12. Key emotional scenes
13. Style consistency notes
14. Risk assessment
15. Rewrite suggestions

## Why UI is secondary

The current priority is not expanding the interface. The core product value is the reasoning pipeline:

```text
Corpus map + author style bible + story bible + unresolved threads
        ↓
Multiple ending route generation
        ↓
Route comparison matrix
        ↓
Hybrid route design
        ↓
Final volume outline
        ↓
Chapter-level outline
        ↓
Scene-level outline
        ↓
Style-sensitive prose drafting only after approval
```

A UI can be added later after the planning workflow is stable.
